# Strategy Rules Removal

**Date**: 2026-04-23
**Status**: Approved, pending implementation plan
**Scope**: Delete the `StrategyRule` / `rule_distiller` / `RuleStore` subsystem in its entirety

## Motivation

The postrun pipeline calls `distill_rules` every `DISTILL_EVERY_N_RUNS` runs.
It compares 5 winning vs 5 losing runs using shallow summaries (character,
archetype, deck preview, top-5 plays, combats W/L, last combat HP delta) and
asks the LLM to extract generic strategy rules. The produced rules are injected
into prompts at 7 sites — 5 in `retriever.py`, 1 in `evolution_engine.py`, and
also flow through `WorkingContext.rule_hints`.

Three pieces of evidence converge to the same conclusion: **the feature is
architectural dead weight**.

### 1. The input is too shallow to differentiate winners from losers

The distill prompt has no traceable signals (per-action damage / block / power
attribution), no deck event trajectory, no relic context, no strategic notes,
no per-decision comparison. Winners reach Act 3, losers die at F20 — the
confounders (floor reached, relics accumulated, deck maturity) dominate any
strategy variable. The LLM is asked to find "what winners did differently"
but the evidence doesn't support that conclusion, so it produces generic
evergreen advice that rehashes seed skills.

### 2. The rule lifecycle is dead code

`StrategyRule.with_verification()` / `with_violation()` exist on the model.
`RuleStore.record_verification()` / `record_violation()` exist on the store.
**Grep confirms zero production callers for either public method.** Rules are
born at confidence 0.5 and stay at 0.5 forever. There is no learning loop. The
lifecycle machinery is a fossil of an earlier design intent that never got
wired up.

### 3. The output space overlaps existing layers

The architectural principle already established (per the non-combat discovery
removal on the same day):

> **Skills = error correction, guides = memory.**

Rules are neither. Their only differentiated role would be "cross-key generic
heuristics" — but that is precisely the job of hand-authored seed skills
(`core_combat.json`, `core_deck_building.json`, `core_map_routing.json`,
`core_rest_decision.json`). Seeds are authored by people who played the game;
rules are generated by an LLM staring at confounded summaries. When the two
would disagree, seeds are right.

## Goal

Delete the entire rules subsystem: the distiller, the store, the model, the
prompt, the counter, the injection sites, the config flag, and its tests.
Leave `memory/rules.json` on disk (orphan file).

Non-goals:

- Any change to seeds, guides, mistake-driven combat skill discovery, or
  guide consolidation.
- Any migration of existing rule data into another surface — rules are
  discarded; the authored seeds already cover the space.
- Purging `memory/rules.json` from the data repo — leave as orphan, out of
  scope for this change.

  Batch API consumers land).

## Data compatibility

| Artifact | Action |
|---|---|
| `memory/rules.json` (sibling repo or local fallback) | Leave on disk. No reader remains; file becomes orphan. |
| `data/skill_distill_count.json` or any `run_count` counter file written by the removed distillation counter | Leave on disk as orphan. Not purged. |
| Legacy in-flight Anthropic batches with `task["type"] == "distillation"` | Silently dropped on receipt — the handler branch is gone. Non-critical. |

## Invariants preserved

1. `WorkingContext` still aggregates combat/route/deck/event/card memories
   and skill hints at decision time. Only `rule_hints` is removed.
2. Guide consolidation counter and cadence unchanged — still fires every
   `CONSOLIDATION_EVERY_N_RUNS` runs.
3. Seeds continue loading and retrieval.
4. Mistake-driven combat skill discovery unaffected.
5. Evolution engine still has access to combat guides, deck guides, event
   guides, skills, and the strategic thread — only rule injection is dropped.

## Risks

1. **`retrieve_working_context` signature change** — removing the
   `rule_store` kwarg is a cross-file signature change. Every caller must be
   updated in the same commit that changes the signature. Grep audit
   required before merge.
2. **`WorkingContext` field removal** — dropping `rule_hints` is likewise a
   cross-file data-shape change. Any prompt formatter that iterates
   `working_context.rule_hints` or checks its length must be updated.
3. **MemoryManager initialization order** — the `self.rules = RuleStore.load(...)`
   call currently happens before `self._init_v2_stores(base)`. Removing it
   must not break ordering assumptions in `_init_v2_stores`.
4. **Counter file reuse** — verify no other feature reads from the same
   counter file path that the distillation counter used.

## Validation plan

  no new failures.

- Short live run: `python -m scripts.run_agent --steps 50 --runs 1` —
  postrun log must not contain `rule_distillation`, `distill_rules`, or
  `record_rules`. `guide consolidation` + `mistake_discovery` + non-combat
  seed scoring still fire.
- Grep audit: `grep -rn 'StrategyRule\|RuleStore\|rule_store\|rule_hints\|rule_distiller\|record_rules\|distill_rules\|should_distill\|RULE_CAPACITY\|DISTILL_EVERY_N_RUNS' src/ tests/ scripts/ config.py` must return zero hits.
